# Telegram Clone System Frontend UI Skill

本 Skill 用于约束 Telegram Clone System 后台管理前端 UI。

项目背景：
- 前端位于 admin/
- 使用 Vue + Vite
- 使用 Element Plus
- 项目是 Telegram 克隆、分发、客服、频道管理后台
- 主要用户是运营人员，不是普通消费者
- UI 优先追求清晰、稳定、高效率、易排查问题

## 一、总体设计原则

1. 后台风格保持简洁、专业、信息密度适中。
2. 不要做花哨动画，不要使用复杂渐变，不要引入大型 UI 库。
3. 优先复用 Element Plus 组件。
4. 所有页面应保持一致的布局、按钮、表格、表单、状态标签风格。
5. 新功能不要破坏旧页面布局。
6. 页面必须适配 1366px 宽度以上的桌面浏览器。
7. 移动端不是第一优先级，但页面不能完全错乱。
8. 所有危险操作必须二次确认。
9. 所有异步操作必须有 loading、成功提示、失败提示。
10. 所有 API 错误要展示可读错误信息，不要只显示 undefined 或 Network Error。

## 二、后台页面标准布局

所有列表型页面采用统一结构：

页面标题
↓
说明文案，可选
↓
筛选区
↓
操作按钮区
↓
数据表格
↓
分页

示例：

- 标题：我的频道
- 说明：统一管理所有目标频道，任务创建时可从这里选择频道。
- 筛选：关键词、状态、分组、绑定 Bot
- 操作：新增频道、批量检测、刷新
- 表格：频道名称、username、chat_id、分组、权限、状态、最后检测时间、操作
- 分页：底部右侧

## 三、表格规范

所有表格统一：

1. 使用 el-table。
2. 表格必须有 loading 状态。
3. 空数据必须显示明确提示。
4. 长文本用 ellipsis，鼠标 hover 可查看完整内容。
5. 时间统一显示 YYYY-MM-DD HH:mm:ss。
6. 状态使用 el-tag。
7. 操作按钮统一使用 text button，不要每行堆很多彩色按钮。
8. 删除、禁用、清空、重置等危险操作必须 el-popconfirm 或确认弹窗。
9. 所有表格高度固定，最多显示约 10 行，超出内容在表格内部滚动，不允许随着数据量持续撑高页面。
10. 表格行内容优先单行展示，长文本使用省略号和 tooltip，避免一行数据占用两行以上高度。
11. 表格列顺序建议：
   - 主要名称
   - 标识字段
   - 类型/分组
   - 状态
   - 时间
   - 操作

## 四、状态标签规范

统一使用以下状态：

enabled：
- 文案：正常
- 类型：success

disabled：
- 文案：已禁用
- 类型：info

error：
- 文案：异常
- 类型：danger

pending：
- 文案：待检测 / 待处理
- 类型：warning

running：
- 文案：运行中
- 类型：success

stopped：
- 文案：已停止
- 类型：info

blocked：
- 文案：已拉黑
- 类型：danger

unknown：
- 文案：未知
- 类型：info

## 五、表单规范

新增/编辑页面优先使用弹窗 Drawer 或 Dialog。

简单表单用 Dialog。
复杂表单用 Drawer。

表单要求：

1. 使用 el-form。
2. 必填项必须有 rules。
3. 保存按钮必须有 loading。
4. 保存成功后关闭弹窗并刷新列表。
5. 保存失败显示后端错误。
6. 编辑时必须回显原始数据。
7. 输入 username 时，如果用户没写 @，前端可自动补 @，后端也要兜底。
8. 不要让用户填写无意义的内部字段。
9. 对 chat_id、bot token、proxy 等敏感字段要有说明。

## 六、筛选区规范

列表页筛选区统一使用 inline form。

常见筛选字段：

- 关键词 keyword
- 状态 status
- 分组 group_name
- Bot bot_id
- 时间范围 date_range

筛选按钮：

- 查询
- 重置

重置必须清空筛选条件并重新加载列表。

## 七、公共组件规范

需要沉淀以下公共组件：

### 1. ChannelSelect

用于选择“我的频道”。

功能：
- 支持单选
- 支持多选
- 支持搜索
- 支持按分组展示
- 默认只显示 enabled 频道
- 异常频道显示警告状态
- 支持 includeDisabled 参数
- 支持 botId 参数筛选指定 Bot 可用频道

显示格式：

频道名称 / @username / 分组 / 状态

### 2. BotSelect

用于选择 Bot。

显示：
- Bot 名称
- Bot username
- 状态

### 3. AccountSelect

用于选择 Telegram 用户号。

显示：
- 账号名称
- username
- phone
- enabled 状态
- session 是否有效，如果后端有返回

### 4. StatusTag

统一状态标签。

### 5. JsonPreview

用于预览复杂 JSON 配置。

### 6. ErrorText

用于展示后端错误信息。

### 7. CopyText

用于复制 chat_id、token、链接等。

## 八、我的频道页面 UI 规范

“我的频道”页面是频道资产库。

页面字段：

- 频道名称
- username
- chat_id
- 类型
- 分组
- 绑定 Bot
- 权限
- 状态
- 最后检测时间
- 备注
- 操作

权限展示方式：

使用小标签或图标：

- Bot 在频道
- 管理员
- 可发帖
- 可编辑
- 可删除
- 可管理话题

不要把权限显示成一大段文字。

操作按钮：

- 编辑
- 检测
- 启用/禁用
- 删除

批量操作：

- 批量检测
- 批量启用
- 批量禁用

## 九、任务页面目标频道选择规范

所有需要选择目标频道的地方，不要再使用普通 input 手动输入。

必须使用 ChannelSelect。

规则：

1. 单目标使用单选。
2. 附加目标、批量目标使用多选。
3. 如果旧任务目标不在 my_channels 中，页面仍要显示原始值，并提示：
   未加入我的频道
4. 保存时优先保存 my_channel_id。
5. 为兼容旧逻辑，也可继续保存 username/chat_id。
6. 多选目标必须能正确回显。

## 十、客服模块 UI 规范

客服后台主要用于配置和只读查看，不做复杂聊天窗口。

页面包括：

- 客服设置
- 客户列表
- 会话记录
- 快捷回复

客服设置页：

字段：
- 客服 Bot
- 客服群 chat_id
- 欢迎语
- 非营业时间回复
- 接收模式 polling
- 测试 Bot
- 获取最近 updates

客户列表：

字段：
- username
- 昵称
- telegram_user_id
- 来源
- 最后消息时间
- 状态
- 是否拉黑
- 操作

会话记录：
- 只读
- 区分 customer / support / bot
- 支持文本、图片、视频、文件记录显示
- 不需要后台直接回复输入框

## 十一、错误告警配置 UI 规范

如果实现错误告警配置页面，放在系统设置中。

字段：

- 是否启用告警
- CONTROL_BOT_TOKEN
- CONTROL_CHAT_ID
- 发送测试通知按钮

说明文案：

控制 Bot 用于发送系统错误通知，例如任务失败、账号掉线、权限异常。
控制 Bot 必须加入错误通知频道，并拥有发送消息权限。

Token 输入框要支持：
- 显示/隐藏
- 清空
- 保存

测试按钮：
- loading
- 成功提示
- 失败显示 Telegram API 错误

## 十二、日志和错误展示规范

所有任务、Bot、频道检测、客服消息发送错误，前端展示时要保留：

- 错误类型
- 错误消息
- 发生时间
- 关联对象
- 可复制原始错误

不要只显示“失败”。

示例：

频道权限检测失败：
Bot 不是频道管理员。
请把 Bot 添加为频道管理员，并授予发帖权限。

Bot API 失败：
403 Forbidden: bot is not a member of the channel chat

账号 session 失效：
账号 session 已失效，请运行 login_account.py 重新登录。

## 十三、加载和空状态规范

所有页面必须有：

1. loading 状态
2. empty 状态
3. error 状态

空状态文案要具体。

不要写：
暂无数据

应该写：
暂无频道，请点击“新增频道”添加你的目标频道。

主要操作：
- primary

普通操作：
- default

危险操作：
- danger

文本按钮：
- 表格行操作使用 link/text 风格

常见按钮文案：

- 新增
- 编辑
- 保存
- 取消
- 删除
- 检测
- 刷新
- 重置
- 查询
- 启用
- 禁用
- 发送测试

## 十五、接口调用规范

前端 API 调用必须统一封装。

要求：

1. 不要在页面里到处直接写 fetch。
2. 新功能新增 api 文件，例如：
   admin/src/api/myChannels.js
3. 所有错误统一处理。
4. 页面内只处理业务提示。
5. 所有保存、检测、删除操作都要 loading。
6. 请求失败显示后端返回 message。

## 十六、文件组织建议

如果当前项目已有结构，优先保持现有风格。

建议结构：

admin/src/views/MyChannels.vue
admin/src/api/myChannels.js
admin/src/components/ChannelSelect.vue
admin/src/components/StatusTag.vue

如果已有 components 或 views 目录，请复用。

## 十七、兼容旧功能

新增 UI 不允许破坏：

- 频道克隆
- 频道分发
- 客服 Bot
- Bot 设置
- 账号管理
- 任务列表
- 模板功能

旧字段仍要可显示，旧任务仍要可编辑。

## 十八、验收标准

每个新增前端功能必须满足：

1. 页面能正常打开。
2. 列表能加载。
3. 新增能保存。
4. 编辑能回显。
5. 删除/禁用有确认。
6. 筛选和重置可用。
7. loading 状态正常。
8. API 报错有明确提示。
9. 页面刷新后数据仍在。
10. Docker 构建前端成功。
11. 线上访问 http://45.76.176.48:5173 正常。
12. 浏览器控制台无明显 JS 报错。

## 十九、给 Codex 的前端开发要求

Codex 修改前端时必须：

1. 先检查现有 admin 项目结构。
2. 优先复用已有样式和组件。
3. 不引入新的大型依赖。
4. 不重构无关页面。
5. 新增页面要加路由和菜单。
6. 新增 API 要和后端接口匹配。
7. 新功能完成后说明改了哪些文件。
8. 如果需要后端接口，必须同时实现或说明依赖。
9. 如果涉及数据库，必须有后端 migration。
10. 完成后给出线上部署命令。
